On-Device AI QNN Internals GPU / HTP Deep Dive May 2026

Inside QNN's Execution Engine —
Threads, Tiles & Tensor Ops

Most engineers call qnn-net-run and move on. But what actually happens between "dispatch" and "output tensor"? This is the layer where real mobile AI performance is won or lost.

The previous post explained who runs your LLM on Snapdragon — CPU for orchestration, GPU for prefill, NPU for decode. This post goes one level deeper: how each processor actually executes a tensor operation. We're talking threads, tiles, wavefronts, shared memory, DMA pipelines — the engineering that makes the difference between 20 tokens/sec and 45.

Everything here is illustrated with animated diagrams. Each one shows a concept playing out step by step — not as an abstract description, but as an actual execution trace.

Section 01

The Problem: A Matrix Multiply Is Deceptively Expensive

Every transformer layer in an LLM is dominated by one operation: matrix multiplication. In a 7B model running on Snapdragon, a single attention projection looks like this: 

// One linear projection in Llama 3.1 7B
input   : [1 × 4096]        ← one token, 4096-dim embedding
weights : [4096 × 4096]     ← Q, K, V, or O projection matrix
output  : [1 × 4096]

// That's 4096 × 4096 = 16.7 million multiply-accumulate ops
// For ONE projection. There are ~224 such ops per decode step.

Done naively — one thread computing one output element — this would take millions of sequential steps. Instead, QNN maps this onto thousands of parallel threads using a technique called tiling. The animation below shows exactly how.

Each tile maps directly to one workgroup on the Adreno GPU. All workgroups run simultaneously — that's the parallelism. The key insight: instead of computing one output element at a time, QNN computes an entire 16×16 block at once using 256 threads working in lockstep.

Why tiles? A 4096×4096 matrix has 16.7M elements. With 256-thread workgroups, that's ~65,000 workgroups — all dispatched simultaneously. The Adreno GPU on Snapdragon 8 Elite can schedule thousands of these concurrently across its compute units.
Section 02

Wavefronts — When 64 Threads Run as One

Inside each workgroup, threads don't execute completely independently. Adreno groups them into wavefronts — sets of threads that execute the exact same instruction simultaneously, just on different data. This is SIMT: Single Instruction, Multiple Threads.

Intuition

A wavefront is like a marching band row. Every musician plays a different note (different data), but they all play on the same beat (same instruction). If one musician needs to stop early, the whole row waits — that's called divergence, and it's expensive.

Section 03

The Shared Memory Trick — DDR Is the Enemy

Here's the bottleneck most engineers don't think about: if every thread independently reads its data from DRAM (main memory), the bandwidth cost is catastrophic. A single matmul tile would cause hundreds of DRAM round-trips — each one taking 50-100 nanoseconds.

QNN solves this with shared local memory (SRAM on the GPU, TCM on the HTP). The entire tile of data is loaded from DRAM once, placed into fast on-chip SRAM, and then all 256 threads in the workgroup reuse it many times.

Section 04

Memory Coalescing — Access Patterns That Kill Performance

Even with shared memory, how threads access it matters enormously. Memory buses on mobile GPUs are designed to handle coalesced access — when adjacent threads read adjacent addresses, the hardware can serve them in a single wide transaction. Scatter them across memory and performance collapses.

Animation 4 — Memory Coalescing: Sequential vs Scattered Access
✓ COALESCED — T0→addr0, T1→addr1, T2→addr2, T3→addr3
→ 1 memory transaction
❌ SCATTERED — T0→addr0, T1→addr7, T2→addr3, T3→addr12
→ 4 separate transactions (4× bandwidth waste)
QNN rearranges tensor layouts (NHWC → blocked) to guarantee coalescing
What QNN does about this: Before dispatching kernels, QNN may transform tensor memory layouts — converting from standard NHWC to blocked formats where channels are packed in groups of 32 or 64, aligned to cache line boundaries. This guarantees coalesced access even for tensors that weren't originally laid out that way.
Section 05

The Decode Problem — Why Arithmetic Intensity Collapses

Everything above applies beautifully to prefill: big matrices, lots of data reuse, high GPU utilization. But during decode — token generation — the workload changes completely. You're multiplying a tiny [1 × 4096] vector against a large weight matrix. The tile tricks still apply, but the ratio of compute to memory access (called arithmetic intensity) plummets.

Animation 5 — Arithmetic Intensity: Prefill vs Decode on the Roofline
The roofline model explains why GPU looks idle during decode

During decode, arithmetic intensity drops below the ridge point — you're in the memory-bound region. The GPU's compute units are starved, waiting on KV cache reads from LPDDR5X. This is why low GPU utilization during decode is not a bug — it's physics. The fix is reducing memory traffic (quantized KV cache, GQA), not adding more compute.

The Snapdragon Profiler trap: If you see GPU utilization at 22% during decode and assume the GPU is idle — you're misreading the data. Fire up the memory bandwidth counter. You'll likely see it pegged at or near the LPDDR5X ceiling of ~77 GB/s.
Section 06

HTP Execution — A Totally Different Animal

The Hexagon Tensor Processor (HTP) is architecturally nothing like the Adreno GPU. Instead of thousands of lightweight threads, it runs a small number of deeply scheduled operations using a hierarchy of specialized compute units:

S
Scalar Core (VLIW)
6–8 hardware threads running control code. Handles branching, loop control, memory management. Like a very lean CPU inside the NPU. Runs compiled C code — not GPU shaders.
V
HVX — Hexagon Vector eXtensions
512-bit or 1024-bit wide SIMD vectors. Handles element-wise ops: activations (SiLU, GeLU), normalization, residual additions. Each HVX instruction processes 64–128 elements simultaneously.
T
HMX — Hexagon Matrix eXtensions (Tensor Accelerator)
The INT4/INT8/FP16 matrix multiply engine. Executes 16,000+ MAC ops per cycle. This is what handles W4 weight projections in decode. Instructions are not publicly documented — QNN generates them internally.
D
DMA Engine + TCM
8MB of Tightly Coupled Memory (TCM) — software-managed on-chip SRAM. The DMA engine loads tiles from DRAM into TCM asynchronously, overlapping with HMX computation. This overlap is the HTP's biggest performance lever.

Without double buffering, HMX must wait for DMA to finish loading before it can compute — leaving it idle roughly half the time. With double buffering, while HMX computes tile A, the DMA engine loads tile B. Then they swap. Hardware is always busy.

Section 07

Operator Fusion — The Biggest Single Speedup

Every kernel launch has overhead: command submission, memory barriers, synchronization. In a transformer layer, a naive implementation would run MatMul → sync → RMSNorm → sync → SwiGLU → sync → next MatMul. Each sync is a barrier where all threads must stop and wait.

QNN's compiler fuses adjacent operators into single super-kernels where possible, eliminating intermediate memory writes and barrier costs entirely.

Section 09

The Hierarchy That Actually Matters

For GPU (Adreno): Performance bottleneck order is usually: memory bandwidth > coalescing > occupancy > FLOPS. Fix bandwidth before adding compute.

For HTP (Hexagon): Performance bottleneck order is usually: DMA throughput > TCM capacity > HMX utilization > operator support. If you're not overlapping DMA and compute, you're leaving 50% performance on the table.

For both: Operator fusion is almost always the highest-ROI optimization. Two synced kernels with one DRAM write between them lose more time to the barrier than either kernel spends computing.
