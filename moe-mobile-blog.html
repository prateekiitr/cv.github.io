Why MoE on Mobile is a Nightmare —
And How Meta Finally Cracked It

Mixture-of-Experts is the architecture behind every frontier LLM. But putting it on a smartphone? That's a completely different problem. Here's the full picture — the constraint, the cache thrashing, and the three engineering moves that made it work.

📅 May 27, 2026 ✍️ Prateek Singh ⏱ 8 min read 📄 Based on MobileMoE (Meta AI, arXiv 2605.27358)

Mixture-of-Experts has become the default architecture for the world's best language models — DeepSeek V3, Qwen3-MoE, Gemini, Grok. The idea is elegant: instead of one massive FFN block per transformer layer, you have many small expert networks, and each token only activates a few of them. You get the capacity of a large model at a fraction of the inference cost.

So the obvious question is: why hasn't this reached your phone?

On-device LLMs — MobileLLM, Gemma 3 1B, SmolLM2, Llama 3.2 1B — are all dense. Every single one. And it's not for lack of trying. The reason is fundamental to how mobile hardware works.

The Core Problem: Mobile is Memory-Bandwidth Bound

On a server GPU, inference is compute-bound. You have teraflops of math horsepower, giant HBM caches, NVLink between GPUs. The bottleneck is arithmetic.

On a smartphone, inference is memory-bandwidth-bound. The CPU or NPU sits largely idle, waiting for weight matrices to arrive from DRAM. The bottleneck is data movement, not math. 

// Dense model — what happens every decode step
Load FFN weights (same every time, may still be in cache)
Compute
Done ✓

// MoE model — what NAIVELY happens every decode step
Router decides: use experts 3, 7, 12, 45
Load Expert 3 weights from DRAM  ← cache miss
Load Expert 7 weights from DRAM  ← cache miss
Load Expert 12 weights from DRAM ← cache miss
Load Expert 45 weights from DRAM ← cache miss
Next token → different 4 experts → repeat
DRAM thrashing every single token ✗

This is the fundamental tension. MoE's whole value proposition — sparse activation — becomes a pathological access pattern on mobile hardware. The router changes which experts are needed every token. SRAM (the fast on-chip cache) is tiny — maybe 4–8 MB. Each expert weight matrix is several MB. You can't keep all 60 experts cached. You can't prefetch because you don't know the next routing decision. The result is guaranteed cache misses on every token.

⚠️ The Fundamental Tension

MoE's sparsity is its strength on servers (less compute). On mobile, that same sparsity becomes a weakness — unpredictable memory access patterns that thrash the tiny SRAM cache on every single token.

Meta's MobileMoE paper (released today, May 27, 2026) is the first work to systematically solve this. They deploy sub-billion-active-parameter MoE models on the Samsung Galaxy S25 and iPhone 16 Pro with 1.8–3.8× faster prefill and 2.2–3.4× faster decode than a comparable dense baseline. Here's exactly how.

THREE SOLUTIONS

How MobileMoE Solved It

01

Token Reordering via Counting Sort

Before any expert computation happens, MobileMoE sorts all tokens by their expert assignment. Every token headed to Expert 3 is grouped together. Every token headed to Expert 12 is grouped together. 

// Before reordering (naive)
Token 1  → Expert 3
Token 2  → Expert 47
Token 3  → Expert 3   ← forces reload of Expert 3
Token 4  → Expert 12

// After counting sort
Expert 3  → [Token 1, Token 3, Token 8, Token 15 ...]
Expert 12 → [Token 4, Token 9 ...]
Expert 47 → [Token 2, Token 6 ...]

// Now process expert-by-expert:
Load Expert 3  ONCE → process ALL its tokens → done
Load Expert 12 ONCE → process ALL its tokens → done
Load Expert 47 ONCE → process ALL its tokens → done

Result: instead of one cache miss per token, you get one cache miss per active expert. With 4 active experts and 60 tokens in a batch, that's 4 loads instead of 60.

02

Grouped MLP — One Big GEMM Instead of Many Small Ones

Here's the second problem that token reordering alone can't fix: MobileMoE's experts are tiny. Each expert FFN in MobileMoE-S is 768×384. Compare that to Mixtral's experts at 4096×14336 — roughly 200× larger.

Small matrix multiplications are horribly inefficient. Mobile hardware (CPU, GPU, NPU) is designed for large dense operations. When you call a tiny GEMM kernel, most of the time is spent on kernel launch overhead and memory access setup — not actual math. Hardware utilization drops to ~20%. 

// Naive: many small GEMMs
Expert 3  weights × its tokens  → tiny GEMM, ~20% util
Expert 12 weights × its tokens  → tiny GEMM, ~20% util
Expert 47 weights × its tokens  → tiny GEMM, ~20% util

// Grouped MLP: stack all active experts into ONE matrix
[ Expert 3  weights  ]
[ Expert 12 weights  ]  ×  all tokens  →  ONE big GMM
[ Expert 47 weights  ]
                              hardware utilization: ~80%

By batching all expert computations into a single grouped matrix multiplication (GMM) kernel, MobileMoE makes the hardware operate at full efficiency. One kernel call, one memory read, all experts computed together.

03

Fused MoE Kernel — Zero Intermediate DRAM Writes

Even with token reordering and grouped GEMM, there's still a problem: the MoE FFN block has 9 separate sub-operations. In a naive implementation, each step writes its output to DRAM and the next step reads it back. 

// Naive: 9 kernel calls, 9 DRAM round-trips
1. Router logits         → write to DRAM
2. Top-k selection       → read/write DRAM
3. Token dispatch        → read/write DRAM
4. Gate projection       → read/write DRAM
5. Up projection         → read/write DRAM
6. SwiGLU activation     → read/write DRAM
7. Down projection       → read/write DRAM
8. Weighted scatter      → read/write DRAM
9. Unpermute tokens      → write to DRAM

// Fused kernel: 1 call, everything stays in SRAM/registers
Route → dispatch → gate+up → SwiGLU → down → scatter
All in one op. Zero intermediate DRAM writes.

XNNPACK (the standard mobile inference backend) has no native MoE operator. Meta built a custom fused MoE operator inside ExecuTorch. Attention layers still use XNNPACK's optimized INT4 dense path. Only the MoE FFN blocks route through the custom op.

💡 Why DRAM Writes Hurt So Much

Mobile SRAM (on-chip cache) is maybe 4–8 MB and extremely fast. DRAM is 50–80× slower. Every intermediate write forces the CPU to stall waiting for data. The fused kernel keeps everything in fast registers and SRAM until the final output — no stalls.

THE BIGGER PICTURE

What This Means for On-Device AI

The three-solution stack — token reordering → grouped GEMM → fused kernel — is not just a MobileMoE trick. It's a blueprint for any sparse model running on memory-bandwidth-constrained hardware.

"Sort tokens first, batch all expert compute together, never leave the fast cache until done."

The paper explicitly lists NPU deployment as unsolved future work. This is the next frontier. Snapdragon's Hexagon DSP, Apple's Neural Engine, Samsung's Exynos NPU — all designed for fixed, static computation graphs. MoE's dynamic routing (which experts activate depends on the input) is fundamentally at odds with how NPUs are compiled.

Solving dynamic routing on NPUs — whether through hybrid CPU+NPU execution, static compilation of all expert paths, or constrained routing with small E — is where the next 3–5× efficiency gains are hiding. The CPU path MobileMoE uses gets you 3.8× over dense. The NPU path could get you another 5–10× on top of that.

🔭 What's Next

The unsolved problems: NPU deployment of MoE (dynamic routing vs. static graphs), task-conditional expert loading (load only the experts relevant to your app), and hybrid MoE+SSM architectures that eliminate KV cache growth at long context while keeping routing efficiency. Each of these is a publishable research direction on its own.
