LLM Inference Quantization Memory Optimization

Everyone's Quantizing the Wrong Half of Their LLM

GPTQ, AWQ, W4, W3 — the weight quantization race is practically over. Meanwhile, the actual memory hog at inference time is sitting quietly in the background, doubling every time context length goes up.

Prateek Singh April 2026 ~10 min read

Here's a question that should bother you more than it does: if your 70B model is quantized to 4-bit and your VRAM is still melting at long context — what exactly is eating it?

The LLM quantization conversation has been almost entirely dominated by weight quantization — W8, W4, W3, GPTQ, AWQ, QuaRot, SpinQuant, the whole lot. And for good reason: weights are big, they live in VRAM, and compressing them gives obvious wins in memory footprint and load time.

The KV cache has been the quieter story — until recently. Google's TurboQuant (which I broke down in an earlier post) was one of the first methods to bring rotation-based quantization specifically to the KV cache, with headline numbers of 6× compression and 8× speedup. This post zooms out: what the problem actually is, why it's different from weight quantization, and the full landscape of techniques tackling it.

But once you actually start generating tokens, the thing eating your VRAM isn't the weights anymore. It's the KV cache.

And almost nobody is talking about it.

What is the KV cache, exactly?

In a transformer, every attention layer computes three projections for each token: Query (Q), Key (K), and Value (V). During autoregressive decoding, when generating token t, you need the K and V tensors from all previous tokens — not just the current one — to compute attention.

Re-computing those K and V projections from scratch at every step would be catastrophically expensive. So instead, you cache them. Every generated token appends new K and V vectors to a growing buffer. That buffer is the KV cache.

The problem is that it grows linearly with sequence length. For a model like LLaMA-3 70B with 80 layers, 8 KV heads, and a head dimension of 128, the math looks like this: 

# KV cache size estimate (FP16)
layers     = 80
kv_heads   = 8       # GQA reduces this from 64
head_dim   = 128
seq_len    = 4096
bytes_fp16 = 2

kv_cache_bytes = 2 * layers * kv_heads * head_dim * seq_len * bytes_fp16
# → ~671 MB for 4096 tokens, just the cache
# → ~10.7 GB at 65K context
# → ~21 GB at 128K context

Scale that to 128K context — increasingly common with models like Mistral, Gemini, Claude, and Command R+ — and you're looking at tens of gigabytes just for the cache. On a single A100 with 80GB VRAM, serving long-context requests at scale becomes a juggling act.

At 128K context, the KV cache is no longer a footnote. It's competing directly with the model weights for the same VRAM budget.

Why not just quantize it like weights?

This is the obvious question. You compress weights to W4 and things mostly work. Why can't you just apply INT8 or INT4 to the KV cache?

You can — but the distribution of KV tensors is fundamentally different from weight tensors, and that changes what works.

The Key and Value tensors are not the same animal

Research across multiple papers — notably KVQuant (Hooper et al., 2024) and KIVI (Liu et al., 2024) — consistently finds that Keys and Values have very different quantization sensitivities.

  • Key tensors exhibit heavy-tailed distributions with persistent outlier channels that appear across tokens. These outliers don't go away with calibration — they're structural.
  • Value tensors are comparatively well-behaved. Their distributions are smoother, and simple per-channel quantization at INT4 introduces minimal degradation.
Key Insight

Treating K and V with the same quantization scheme is a mistake. Keys need more bits or more careful handling. Values can tolerate aggressive compression. Most naive KV quantization ignores this asymmetry entirely.

The outlier channel problem

A small fraction of channels in the Key cache carry disproportionately large magnitudes — and they're the same channels across different tokens. This means a per-token quantization approach (where you compute scale/zero-point independently per token) misses them. Per-channel quantization handles this better but adds overhead during cache writes.

KVQuant addresses this directly by isolating outlier channels and storing them at higher precision, while compressing the rest to INT4. The result is a non-uniform mixed-precision cache that reduces average memory footprint while preserving the structure that attention actually depends on.

KIVI — asymmetric precision done right

KIVI's central observation is that Value tensors tolerate INT2 quantization far better than Keys do. Their solution: store V at 2-bit with a grouped scheme, and K at 4-bit. They also keep a small "residual cache" — the most recent tokens in full precision — which prevents quality collapse at the beginning of generation where statistical estimates are noisy.

From the literature

KIVI reports near-lossless performance on LLaMA-2 at 2-bit Values + 4-bit Keys, with up to 2.35× memory reduction in practical serving scenarios.

KVQuant — going to 3-bit with surgical precision

KVQuant targets sub-4-bit quantization and actually gets there without catastrophic quality loss. The key tricks: pre-RoPE quantization of Keys (before rotary embeddings are applied, the distribution is more uniform), non-uniform quantization using learned codebooks calibrated on a small dataset, and channelwise outlier handling.

The result is a 3-bit average cache that, on tasks like LongBench, maintains quality close to FP16 — which at this bit-width is genuinely impressive.

Rotation-based smoothing

This is an area borrowed directly from weight quantization: methods like QuaRot and SpinQuant apply random orthogonal rotations to activations to redistribute outliers before quantization. The same idea applied to KV tensors (pre-quantization rotation) meaningfully reduces the dynamic range problem in Key channels.

TurboQuant is the most prominent recent entry in this family — a rotation-based KV cache quantization method from Google achieving 6× compression with minimal accuracy loss. I covered it in depth separately; it pairs well with KIVI and KVQuant as the three techniques actually pushing the sub-4-bit KV frontier.

This is an active research area, and the results are promising — particularly for sub-4-bit Key quantization where outliers are the limiting factor.

Architectural changes that reduce the problem at the source

It's worth noting that some of the KV cache pressure is being addressed architecturally, not just algorithmically.

Grouped Query Attention (GQA)

GQA, used in LLaMA-3, Mistral, and Gemma, reduces the number of KV heads (often from 32 or 64 down to 8) while keeping full query heads. This directly shrinks the KV cache by a factor equal to the grouping ratio — before any quantization is applied. At 8 KV heads vs 64 query heads, you get an 8× reduction in cache size from architecture alone.

Multi-head Latent Attention (MLA)

DeepSeek V2/V3 took this further with MLA — a low-rank compression of the KV projection that reduces the per-token cache size by storing a compressed latent vector instead of full K and V tensors. This is fundamentally different from post-hoc quantization; it bakes cache compression into the attention mechanism itself.

Worth noting

MLA and GQA don't eliminate the need for KV quantization — they just shift the problem. At 128K+ contexts, even compressed KV representations become large enough that quantization on top of architectural compression is still valuable.

The practical considerations nobody talks about

Quantization during writing vs. reading

KV cache quantization happens in the hot path of inference — every token generation involves writing new K/V entries and reading the entire cache for attention. The quantize-on-write / dequantize-on-read overhead needs to be cheap enough to not erase the throughput gains from reduced memory bandwidth. This is where kernel-level implementation matters enormously, and why most academic proposals don't translate directly to wall-clock speedups without careful engineering.

Calibration data dependency

Methods like KVQuant that use non-uniform quantization require calibration on representative data. Unlike weight quantization where you calibrate once at model load time, the optimal KV quantization can vary with input domain. A cache quantized for code generation might behave differently for long legal documents. This is a practical deployment consideration that's often glossed over.

Interaction with speculative decoding

Speculative decoding maintains a draft model's KV cache alongside the target model's. KV quantization applied to both introduces compounding approximation errors in the verification step. This hasn't been systematically studied yet and is an open question.

Where this is heading

KV cache quantization is graduating from a research curiosity to a production necessity. A few signals:

  • vLLM and TGI are actively integrating INT8 and FP8 KV cache support. FP8 is attractive because it's hardware-native on H100s and avoids the calibration complexity of INT4.
  • Context windows are only getting longer. 1M-token models aren't theoretical anymore. At that length, even with GQA, the KV cache dwarfs the weights.
  • The INT4 KV frontier is still rough. Getting clean results below 4 bits across diverse tasks and model architectures without per-model tuning is an unsolved problem.
  • Joint weight + KV quantization is underexplored. Most papers optimize one in isolation. Interactions between quantized weights and quantized activations (the KV cache is an activation, not a weight) are non-trivial.
The open question

Can you push KV cache quantization to INT3 or below — matching the frontier of weight quantization — without either per-task calibration or architectural changes? That answer doesn't cleanly exist yet.
