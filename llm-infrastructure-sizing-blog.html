Sizing an LLM Deployment: The Arithmetic Behind Storage, VRAM, and Concurrency

Four numbers decide whether a GPU fleet is right-sized or quietly wrong: model storage, inference memory, concurrency, and GPU count. Here's how each one is actually derived from architecture and workload — not guessed from a spreadsheet someone copied three jobs ago.

Open the interactive calculator →

Every capacity plan I've reviewed starts the same way: someone estimates how much VRAM a model needs, picks a GPU, and multiplies by a headcount that feels safe. Most of the time it's close enough. Occasionally it's off by 2–3×, in either direction — over-provisioned racks sitting half-idle, or a launch that pages the on-call at the first real burst of traffic.

The gap isn't sloppiness. It's that the arithmetic has four distinct parts, each borrowed from a different discipline — transformer architecture, queueing theory, systems memory management — and it's rare to see all four written down together. This post does that: storage, inference memory, concurrency, and fleet sizing, in order, with the formulas that actually produce each number.

01 · Storage

Model Storage: Parameters Times Precision

This part really is just arithmetic. A parameter stored at a given precision takes a fixed number of bits, and the model file is the parameter count times that width, converted to bytes:

Storage = Parameters × (Bits per Parameter / 8)
// FP32 = 4 bytes/param · FP16/BF16 = 2 · INT8 = 1 · INT4 = 0.5
Model Params FP16 INT8 INT4
Mistral 7B7.3B14.6 GB7.3 GB3.65 GB
Llama 3 8B8B16.0 GB8.0 GB4.0 GB
Qwen 2.5 32B32.5B65.0 GB32.5 GB16.3 GB
Llama 3 70B70B140.0 GB70.0 GB35.0 GB

The only place this trips people up is precision mismatch between disk and serving — a checkpoint stored in FP16 doesn't shrink on disk just because you plan to quantize it at load time. Disk size reflects what you store; VRAM size reflects what you actually load and run, which is the next, more interesting number.

02 · Inference Memory

VRAM Is One Budget, Not a Sum of Independent Guesses

For a GPU-resident deployment, everything the model needs to run a forward pass — weights, KV cache, transient activations, and framework overhead — competes for the same pool of VRAM on the same card. Host RAM is a separate, much smaller concern (OS, orchestration process, a loading buffer), not a second full copy of the model. Four terms make up the VRAM budget, and three of them are commonly estimated wrong.

Weights

Same formula as storage, evaluated at serving precision — this is the one term that's just a lookup.

KV cache

This is where context length and architecture actually enter the picture. The cache holds a key and value vector per token, per layer, for every layer simultaneously — unlike activations, it can't be freed as the forward pass moves on, because generation needs all of it for every future token:

KV cache = 2 × layers × kv_heads × head_dim × seq_len × batch × bytes/elem
// the leading 2 covers K and V; head_dim = hidden_dim / attention_heads
The detail that's easy to miss: the formula uses kv_heads, not total attention heads. Grouped-query attention — used in Llama 3, Mistral, and most current open models — shares key/value projections across groups of query heads, so kv_heads is a fraction of the total head count. Llama 3 8B has 32 attention heads but only 8 KV heads. Sizing the cache off total heads overstates it by 4×, on that model alone.

Activation memory

The forward pass needs working memory for QKV projections, attention scores, and the MLP intermediate — real, and usually the most over- or under-estimated term in a back-of-envelope calculation, because it gets treated like the KV cache when it behaves nothing like it.

Activations ≈ prompt_tokens × hidden_dim × bytes/elem × c  // c ≈ 8, one layer's peak
Not multiplied by layer count. Inference executes layers sequentially and frees each layer's activations before the next one starts — there's no backward pass to retain them for. Training has to keep every layer's activations around for backprop, which is where the "multiply by depth" instinct comes from, and it's wrong here. This formula also assumes flash-attention-style serving, so it skips the O(seq_len²) attention-score term a naive implementation would materialize — at very long context on a non-flash engine, treat it as a floor, not a ceiling.

Fixed overhead

A CUDA context reserves roughly a gigabyte per process before a single model weight loads, and serving engines hold back a fragmentation margin — commonly 10–15% — so a burst in KV cache or activation size doesn't fail an allocation mid-request. Both are small next to weights and KV cache, but they're not zero, and skipping them is why a calculation that "fits exactly" sometimes doesn't in production.

weights: lookup KV cache: scales with context × kv_heads activations: one layer, not all layers overhead: ~1 GB + 10–15% margin
03 · Concurrency

What "Concurrent Users" Actually Means

The KV cache formula above needs a batch size — how many sequences are resident in memory at once. The instinct is to compute this as users times request rate. That number is real, but it isn't concurrency; it's arrival rate, the rate at which new work shows up. Concurrency is a different quantity, and queueing theory already has the formula for it — Little's Law, one of the oldest results in the field:

L = λ × W
// L = avg. requests resident in the system · λ = arrival rate (req/s) · W = avg. time-in-system (s)

λ comes from your usage pattern — users times requests per user per second. W is how long a request actually takes to serve, which is itself prefill time plus decode time:

W = (prompt_tokens / prefill_tok_s) + (output_tokens / decode_tok_s)
Worked example. 1,000 users, each sending 12 requests/hour, 500 prompt tokens and 500 output tokens per request, 3,000 tok/s prefill and 45 tok/s decode per stream:

λ = 1000 × 12 / 3600 = 3.33 req/s
W = 500/3000 + 500/45 = 0.17 + 11.11 = 11.28 s
L = 3.33 × 11.28 = 37.6 → batch of 38 concurrent sequences

That's the number the KV cache formula should use as batch — not 3.33. Sizing memory off the arrival rate alone assumes every request is served instantaneously, which undersizes the cache by roughly the average latency of a request, and gets worse the longer generations run.

04 · Fleet Sizing

From One Replica to a GPU Count

With the per-sequence costs and the required concurrency both in hand, sizing the fleet is two separate questions answered in order — how many cards does it take to even hold the weights, and then how many of those units does the load require:

  1. 1️⃣
    Parallelism floor. Cards needed just to hold the weights: ceil(weight_bytes / usable_vram_per_card), with a small margin held back for context/fragmentation. This is the minimum footprint of one replica, before a single request is served.
  2. 2️⃣
    Headroom per replica. Whatever VRAM is left on that card set after weights and fixed overhead, divided by the KV-cache-plus-activation cost of one sequence, gives the max concurrent sequences one replica can actually hold.
  3. 3️⃣
    Replicate for load. Divide the required concurrency L by that per-replica capacity, round up — that's how many replicas you need.
  4. 4️⃣
    Total GPUs = replicas × cards per replica. Two multiplications, both grounded in a real memory budget rather than a single "GPUs ≈ concurrent users / some constant" heuristic.

Change the model to Llama 3 70B on 24 GB cards and the same four steps still apply — they just resolve differently: seven cards to even hold the weights (the parallelism floor), roughly 18 concurrent sequences per replica once weights and overhead are reserved, and three replicas to cover the same 38-sequence load. Twenty-one GPUs, derived the same way as the single-GPU case, not from a different rule of thumb for "big models."

Takeaway

The Short Version

Storage is parameters times precision — the easy one. VRAM is weights, KV cache, activations, and overhead sharing one pool, where the KV cache scales with context length and the true KV-head count, and activations scale with one layer, not the whole network. Concurrency is Little's Law, L = λ × W, not the arrival rate on its own. And the fleet size is a memory-bound floor for the weights, replicated until it covers that concurrency. Four numbers, four short derivations, no napkin required.
