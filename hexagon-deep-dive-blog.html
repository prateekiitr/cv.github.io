Hexagon NPU HVX · HMX Silicon Deep Dive Qualcomm May 2026

Inside Hexagon —
Qualcomm's NPU from
Silicon to Inference

The Hexagon NPU is not just an "AI accelerator." It's a DSP, a 1024-bit vector engine, a systolic tensor processor, and a DMA controller — all working in concert. This is how it actually works.

The previous two posts explained who runs your LLM on Snapdragon, and how thread execution and operator fusion work. Now we go deeper still — into the Hexagon processor itself. This is the silicon that Qualcomm has been quietly refining since 2006, and in the Snapdragon 8 Elite's V75 generation, it is one of the most sophisticated NPU architectures shipping in any consumer device.

Understanding it at this level is what separates engineers who tune QNN configs from engineers who understand why certain tile sizes, quantization formats, and operator orderings produce dramatically different results.

Section 01

What Hexagon Actually Is

Most documentation describes Hexagon as Qualcomm's "AI processor." That description is technically accurate and practically useless. Here is what Hexagon V75 (Snapdragon 8 Elite) actually contains on-die:

1
Scalar Core — VLIW processor
A Very Long Instruction Word CPU with 6–8 hardware threads. Runs control flow, memory management, loop scheduling. Think of it as a lean embedded CPU inside the NPU — handles everything that isn't pure tensor math.
2
HVX — Hexagon Vector eXtensions
1024-bit SIMD vector units (two per V75 core). Processes 128 INT8 values — or 64 INT16 values — per instruction. Handles normalization, activations (SiLU, GeLU, RoPE), residual additions, and anything element-wise.
3
HMX — Hexagon Matrix eXtensions
The dedicated tensor accelerator. Executes INT4, INT8, and FP16 matrix multiplications at peak TOPS. This is the unit that handles every weight projection in a transformer during decode. Its microarchitecture is not publicly documented — QNN generates its instructions internally.
4
DMA Engine + 8MB TCM
A multi-channel DMA engine connected to 8MB of Tightly Coupled Memory — software-managed on-chip SRAM. The DMA engine loads weight and activation tiles from LPDDR5X directly into TCM, overlapping with HMX computation. This overlap is Hexagon's biggest performance multiplier.
History note: Hexagon V1 shipped in 2006 as a pure audio DSP. V65 (2018) added HVX. V68 (2020) added the first HMX prototype. V73 (Snapdragon 8 Gen 2) brought native INT4. V75 (8 Elite) doubles HMX throughput and introduces micro-tile inferencing. The architecture you see today is 18 years of mobile-specific iteration.
Section 02

VLIW — The Scalar Core That Schedules Everything

The scalar core runs VLIW — Very Long Instruction Word architecture. Unlike a modern CPU that uses out-of-order execution hardware to find parallelism at runtime, VLIW puts that burden entirely on the compiler. Every instruction packet can contain up to 4 operations scheduled in parallel by the compiler at compile time.

Intuition

A standard CPU is like a chef who improvises during service — reads the situation, reorders tasks on the fly. A VLIW processor is like a chef following a microsecond-perfect prep schedule written the day before. Less flexible, but dramatically simpler silicon — no reorder buffer, no branch predictor, no out-of-order logic. That silicon area gets reallocated to HVX and HMX instead.

// VLIW packet: 4 operations execute simultaneously
{ 
  r0 = memw(r4)        // load from memory
  r1 = add(r2, r3)      // integer add
  p0 = cmp.gt(r5, #0)  // compare
  r6 = r7               // register move
}
// All four execute in one clock cycle
// Compiler verified no data hazards exist

The Hexagon compiler (part of QNN's toolchain) statically schedules HVX and HMX instructions across VLIW slots. This means the compiler, not hardware, decides when to issue vector loads, when to fire HMX operations, and when to stall. Getting this scheduling right is a major part of what QNN's kernel library does.

Section 03

HVX — 1024-Bit Vector Engine

HVX is Hexagon's SIMD engine. Each HVX unit operates on 1024-bit registers — which means in a single vmpy (vector multiply) instruction, it processes 128 pairs of INT8 values simultaneously. V75 has two HVX units per core, doubling that throughput.

HVX handles the operations that HMX can't: element-wise functions (SiLU, GeLU, ReLU), RoPE rotary embeddings, RMSNorm reduction, softmax, residual additions. These aren't matrix multiplies — they're per-element transformations on activation vectors. HVX processes them 128 elements at a time.

Key HVX instructions for LLMs: vmpy (vector multiply), vadd / vsub (residual add), vmem (vector load/store), vdeal / vshuff (data rearrangement for layout transforms), vmax / vmin (activation clamping). RoPE embedding rotations map directly to HVX complex multiply sequences.
Section 04

HMX — The Tensor Accelerator

HMX is where the bulk of LLM compute lives. It is a dedicated matrix multiply engine — closer in spirit to a systolic array than a GPU's SIMD cores. Instead of dispatching thousands of independent threads, HMX moves data through a structured array of multiply-accumulate units in a coordinated wave. Weights flow in one direction, activations in another, and partial sums accumulate as they travel.

INT4 Mode — Why It Matters

In INT4 mode, HMX processes 4-bit weights against 8-bit activations (W4A8 — the dominant quantization format for on-device LLMs). The hardware unpacks two INT4 weights from each byte, scales them, and feeds them into INT32 accumulators. This doubles effective throughput compared to INT8: same silicon area, twice the operations per cycle.

Section 05

INT4 Packing — How Two Weights Share One Byte

W4 quantization stores two weight values per byte — the upper nibble (bits 7–4) and lower nibble (bits 3–0). Before HMX can use them, they must be unpacked, de-quantized using per-channel scale factors and zero points, and fed into the accumulator pipeline. The animation below shows this exactly.

W4A8 is the hardware sweet spot: Weights are INT4 (4× memory savings vs FP16), activations stay INT8 (preserving dynamic range that INT4 activations would lose). The HMX INT32 accumulator is wide enough to prevent overflow from accumulated INT4×INT8 products. This is exactly what GPTQ, AWQ, and QuaRot all target.
Section 06

TCM — 8MB of Software-Managed SRAM

Unlike a GPU's L1/L2 caches — which are hardware-managed and transparent to the programmer — Hexagon's TCM (Tightly Coupled Memory) is software-managed. QNN's compiler explicitly decides what goes in TCM, in what order, and when. There is no hardware eviction policy. If QNN puts something in TCM, it stays there until QNN removes it.

This is a significant advantage for LLM workloads: hardware caches don't know which data will be reused; the compiler does. For a specific tile size and sequence of operations, QNN can guarantee zero cache misses on the weight tiles that matter most.

TCM banking: The 8MB TCM is divided into banks. QNN must schedule tensor loads to avoid bank conflicts — two simultaneous accesses to the same bank cause a stall. For weight matrices laid out in row-major order, QNN may apply a banking-aware transpose during model compilation to guarantee conflict-free access patterns at runtime.
Section 07

The DMA Engine — Qualcomm's Secret Weapon

The DMA engine in Hexagon is not a simple memcpy accelerator. It is a multi-channel, scatter-gather capable data mover that can simultaneously transfer weight tiles from different memory regions while the HMX is computing. This overlap — DMA loading the next tile while HMX computes the current one — is where Hexagon earns its performance-per-watt advantage.

Scatter-gather DMA means the engine can load non-contiguous tensor regions — for example, every 4th row of a weight matrix for a strided operation — without CPU involvement. This matters for attention mechanisms where the KV cache layout can be non-contiguous after paged allocation.

Section 08

How QNN Compiles a Transformer Layer to Hexagon

When you run a W4A8 quantized Llama 3.1 7B through QNN targeting the HTP backend, here is the compilation path for a single FFN (feed-forward network) block:

1
Graph → Subgraph Partitioning
QNN splits the ONNX/QDQ graph into HTP-compatible subgraphs. Operations with unsupported data types or shapes are marked for GPU or CPU fallback. The partitioner estimates per-subgraph latency to find the optimal split boundary.
2
Tiling Decision
For each matmul, QNN's compiler selects a tile size that fits within the 8MB TCM — while maximizing HMX utilization. For a [1×4096]×[4096×4096] decode matmul, this typically means tiling along the output dimension to keep weight tiles resident in TCM across multiple activation vectors.
3
Operator Fusion at HMX Level
Adjacent operators are fused into single HMX passes where possible: MatMul + dequant + bias add can execute as one HMX operation. RMSNorm immediately following a projection is fused with HVX instructions scheduled between HMX outputs — no intermediate DRAM write.
4
DMA Scheduling + Double Buffering
The compiler generates a DMA schedule that prefetches the next weight tile into TCM buffer B while HMX computes the current tile from buffer A. The swap is statically scheduled — no runtime decision required.
5
VLIW Instruction Packing
The final scalar control code is packed into VLIW packets. DMA initiation, HMX launch, HVX element-wise ops, and scalar loop counters are scheduled into parallel slots. The compiler verifies no data hazards exist across all slots before emitting the binary.
Section 09

Power Architecture — Why Hexagon Wins at Battery Life

Hexagon runs on a dedicated power island — it can be powered up independently of the CPU and GPU. This means for a sustained background AI task (wakeword detection, continuous health monitoring, on-device agent), the SoC can keep Hexagon active while the Oryon CPU is in deep sleep and the Adreno GPU is completely off.

V75's micro-tile inferencing takes this further: instead of loading the full model and running it continuously, Hexagon breaks inference into small bursts — loading a tile, computing it, writing the result, and entering a low-power state between bursts. This dramatically reduces average power while maintaining acceptable latency for non-real-time tasks.

Section 11

The Full Picture — From Token to Silicon

A single decode step on Hexagon:

1. CPU hands QNN a [1×4096] activation vector for the current token.

2. QNN's runtime triggers the pre-compiled HTP dispatch graph. No recompilation.

3. DMA Channel 0 loads the Q-projection weight tile (INT4, packed 2-per-byte) from LPDDR5X into TCM buffer A.

4. HMX fires: W4A8 matmul of the activation vector against the weight tile. INT32 accumulators collect partial sums.

5. Simultaneously, DMA Channel 1 prefetches the K-projection weight tile into TCM buffer B.

6. HMX dequantizes outputs via per-channel scale factors. HVX applies RoPE rotation to Q and K vectors (complex multiply, 1024-bit lanes).

7. Buffers swap. HMX processes K-projection while DMA loads V-projection. Zero stall.

8. HVX computes attention scores, applies causal mask, runs softmax. HMX computes V-projection simultaneously.

9. Output vector returned to CPU for sampling. Total time: ~3–5ms at 30 tok/sec.
