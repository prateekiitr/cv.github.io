CPU vs GPU vs NPU —
Who Really Runs
Your LLM on Mobile?

Ask most engineers "which processor runs the AI?" and you'll get a confident but incomplete answer. LLM inference on Snapdragon isn't owned by one processor — it's a carefully orchestrated relay race. Here's how it actually works.

If you've ever profiled an LLM running on a Snapdragon device, you know the truth is far messier — and far more interesting — than "it runs on the NPU." Token generation on modern mobile chips is a tightly choreographed handoff between the CPU, the Adreno GPU, and the Hexagon NPU, each taking the baton at different moments in the generation pipeline.

Understanding this choreography is what separates teams that get 15 tokens/sec from those who ship 45. This post walks through each processor's role, how runtimes make dispatch decisions, and practical guidance for on-device LLM products that hold up in production.

Section 01

The Three Processors — A Mental Model

Modern Snapdragon SoCs are not single chips — they are complete computing platforms on one die. On the Snapdragon 8 Elite, you get the Qualcomm Oryon CPU, the Adreno GPU with its new sliced architecture, and the Hexagon NPU — all sharing a unified memory subsystem and communicating over high-bandwidth interconnects.

Qualcomm Snapdragon 8 Elite — full SoC block diagram showing Oryon CPU cluster, Adreno GPU, Hexagon NPU, X80 modem, and LPDDR5X memory subsystem. Source: Qualcomm
Snapdragon 8 Elite — Oryon CPU internal architecture. 2 Prime cores @ 4.32 GHz + 6 Performance cores @ 3.53 GHz, 12MB L2 cache per cluster, Instant Wake hardware. First custom mobile CPU since Krait. Source: Qualcomm / Android Authority

Each processor has a fundamentally different design philosophy. The CPU optimizes for latency on a single thread. The GPU optimizes for throughput across thousands of parallel threads. The NPU optimizes for efficiency — maximum integer tensor operations per milliwatt. LLM inference needs all three.

Intuition

Think of it like a film crew. The CPU is the director — decides what happens and when. The GPU is the cinematography team — does the heavy parallel work at scale. The NPU is the specialized VFX studio — executes a narrow class of tasks (quantized inference) with unmatched efficiency. Great films need all three.

Key spec: The Snapdragon 8 Elite's Hexagon NPU delivers 45 TOPS — 45% faster and 45% better performance-per-watt than Snapdragon 8 Gen 3. But raw TOPS don't win LLM inference. Memory bandwidth is equally decisive.
Section 02

Prefill vs Decode — Two Radically Different Workloads

Before mapping operators to processors, you need to understand that LLM inference has two distinct phases — and they stress your hardware in completely opposite ways.

P
Prefill — Compute-Bound
All input tokens are processed simultaneously. Massive matrix multiplications, full-context attention, high parallelism. GPU utilization spikes. The Adreno Vulkan compute pipeline dominates here. Bottleneck: FLOPS.
D
Decode — Memory-Bandwidth-Bound
One token generated at a time, autoregressively. Each step loads the full KV cache from memory. Compute intensity drops dramatically. Even with 45 TOPS available, the system is waiting on memory reads — not compute cycles. Bottleneck: bandwidth.
Most common optimization mistake: Over-investing in compute TOPS and ignoring memory bandwidth. Decode speed is almost entirely determined by how fast you read the KV cache — not how fast you multiply matrices. 
// Prefill vs Decode — characteristic workload profile

Prefill  → batch_size : N tokens (full prompt)
          → operation  : large matmuls, full attention
          → bottleneck : FLOPS  → GPU preferred
          → utilization: HIGH

Decode   → batch_size : 1 token per step
          → operation  : thin matmuls + KV cache reads
          → bottleneck : bandwidth → NPU preferred (W4)
          → utilization: appears LOW, but latency is HIGH
Section 03

Who Does What — The Operator-Level Picture

CPU — The Orchestrator

The Oryon CPU runs the runtime (QNN, llama.cpp, MLC-LLM), tokenizes input, schedules kernel dispatches, manages memory allocation, and handles all irregular control flow that GPUs and NPUs cannot express. On Snapdragon 8 Elite, Oryon cores run at up to 4.47 GHz, providing the single-thread responsiveness critical for fast scheduling.

GPU — The Parallel Engine

The Adreno GPU, running Vulkan compute shaders, is where the heavy transformer arithmetic lives. Large weight matrix multiplications, multi-head attention (softmax(QKᵀ/√d)·V), RMSNorm, SwiGLU activations, and KV cache projections are all natural GPU workloads. Vulkan's low driver overhead and fine-grained synchronization make it particularly effective for the rapid kernel-launch cadence of transformer layers.

Qualcomm Hexagon NPU architecture overview — DEEP 2024 session. Block-level layout of scalar, vector, and tensor accelerators within the Hexagon compute hierarchy. Source: Notebookcheck / Qualcomm

NPU — The Efficiency Champion

The Hexagon NPU is purpose-built for quantized integer inference. Its Tensor Accelerator handles INT4 and INT8 operations — the core compute primitive of modern quantized LLMs. The Hexagon's micro-tile inferencing breaks networks into small tiles, eliminating memory traffic across 10+ layers and maximizing scalar/vector/tensor accelerator utilization simultaneously. At W4, the NPU delivers dramatically better performance-per-watt than the GPU.

Section 04

The Hybrid Reality — It's a Relay Race

No single processor dominates production mobile LLM inference. Modern runtimes — QNN, llama.cpp (Vulkan backend), MLC-LLM — implement dynamic operator dispatch, routing each layer to whichever backend is optimal for that operation at that moment.

1
CPU Tokenize & Schedule
Tokenize input, prepare buffers, compile and schedule the kernel dispatch graph.
2
GPU Prefill Pass
Embed tokens, run full-context multi-head attention via Vulkan compute shaders. GPU utilization spikes here.
3
NPU W4 Projection Matmuls
Weight projection layers (Q, K, V, O, FFN) dispatched to Hexagon INT4 Tensor Accelerator.
4
GPU Normalization + Activation
RMSNorm and SwiGLU fused into a single Vulkan kernel to minimize synchronization barriers.
5
NPU Sustained Decode
Each decode step's weight ops run on Hexagon. Micro-tile inferencing keeps power stable through long generations.
6
CPU Sample + Detokenize
Top-k/top-p sampling on the logit distribution, then detokenize and return the output string.
QNN backend strategy: QNN exposes three backends — HTP (Hexagon NPU), GPU (Adreno Vulkan), and CPU (Oryon). For a typical W4-quantized 7B model, HTP handles projections, GPU handles attention, and CPU handles sampling. The runtime selects backends per-subgraph based on profiled latency.

Quantization Changes the Entire Equation

Research Note

At W3 and below, rotation-based quantization methods (SpinQuant, QuaRot) that apply orthogonal transforms before quantization currently outperform pure INT approaches — because they preserve activation distributions that both the NPU and GPU can process more efficiently, at the cost of a small dequantization step.

Section 06

Latency, Throughput, and the Thermal Wall

T
Time to First Token (TTFT)
Dominated by prefill speed. Long prompts = longer prefill. GPU-accelerated prefill with fused Vulkan kernels is the primary lever. Users feel TTFT as responsiveness — it's your first impression metric.
S
Tokens per Second (Decode Speed)
Dominated by memory bandwidth. NPU with W4 reduces the working set, cutting bandwidth pressure. But thermal throttling can drop this 30–50% after sustained generation — a factor that benchmarks almost never capture.
W
Power & Thermals
NPU is non-negotiable for production. Hexagon's dedicated power rail and micro-tile inferencing sustain inference at dramatically lower wattage. For an always-on agent or background assistant, GPU-only inference is simply not shippable.
The thermal trap: In a 60-second benchmark, GPU delivers the highest peak tokens/sec. In a 5-minute user session, NPU wins — because the GPU's thermal envelope triggers SoC throttling, while the NPU holds steady. Always build a sustained-load test harness before publishing performance targets.

Section 08

What This Means for Developers

1. Use W4 quantization as your baseline

W4 is the sweet spot for the Hexagon NPU's INT4 hardware. GPTQ, AWQ, and rotation-aware approaches like QuaRot deliver good quality at W4 on 7B-scale models. Start here, then evaluate whether W3 or W8 is a better fit for your quality/speed tradeoff.

2. Use QNN HTP for decode, Adreno GPU for prefill

Configure subgraph-level backend assignment: HTP for weight matrix ops during decode, Adreno GPU for attention during prefill. Let the runtime profile which mapping wins for your specific model architecture, then lock it in for production.

3. Never benchmark with burst load

Build a sustained-load test harness and measure under realistic session durations — not 60-second sprints. Thermal throttling at the 2-minute mark is one of the most common sources of production surprise in mobile LLM deployments.

4. Watch your KV cache footprint

In decode, KV cache reads dominate bandwidth. GQA (Grouped Query Attention) and sliding window attention are your primary levers for controlling this bandwidth footprint without sacrificing quality.

5. The future: compiler-level NPU + GPU co-optimization

Qualcomm's Hexagon MLIR stack is moving toward tighter NPU–GPU cooperation — shared memory pools, fused operator graphs spanning both backends, and compiler-driven co-optimization. The era of manually routing operators to backends is ending. Understanding the principles here is what lets you guide it when it doesn't.

The bottom line: The CPU vs GPU vs NPU question doesn't have a single right answer — it has a right answer for each operator, each phase, and each thermal state. The engineers who ship great on-device AI are those who stop asking "which processor runs my model" and start asking: which processor runs this operator, at this moment, at this power budget?
